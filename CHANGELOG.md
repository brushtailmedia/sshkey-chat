# Changelog

## [Unreleased]

### Security
- **Phase 21 item 3 ŌĆö edit-signature canonical-form fix (protocol surface only).** The `edit` / `edit_group` / `edit_dm` verbs now use distinct msgID-bound signature canonical forms to prevent a compromised server from replaying a sender's past valid send signatures across different msgIDs (which would let a hostile server rewrite history via Phase 15's `edited` broadcasts that overwrite existing rows in local client DBs). Server-side code is **unchanged** ŌĆö signatures remain opaque pass-through blobs the server does not verify or interpret. The full implementation lives in sshkey-term (new `SignRoomEdit` / `SignDMEdit` / `VerifyRoomEdit` / `VerifyDMEdit` in `internal/crypto`, edit-path send sites updated, client-side verify-or-drop wired into the edit receive handlers). `PROTOCOL.md` "Message Signatures" section documents all four canonical forms (send ├Ś 2, edit ├Ś 2) and the "Message Editing" section documents the client-side verify-or-drop contract. Breaking protocol change for edit signatures ŌĆö zero users per the no-live-users policy, so no compat shim ships. See sshkey-term `CHANGELOG.md` for the full client-side implementation detail.

### Added
- **Phase 19: Server backup & restore.** Server-side snapshot backups + retention + a stop-and-restore CLI. SQLite Online Backup API (via `modernc.org/sqlite`'s `(*conn).NewBackup`) snapshots every DB while the server keeps serving clients ŌĆö no downtime, no client disconnects. Tarballs bundle SQLite DBs + attachment blobs + `audit.log` + `pending-keys.log` + `host_key` + `server.toml` (config files default-included; opt out via `[backup].include_config_files`). `rooms.toml` deliberately not included (seed file, ignored when `rooms.db` exists). See `docs/ops/backup_restore.md` for the operator runbook.
  - **Scheduled backups default-on.** `[backup] enabled = true` in the reference `server.toml`. Cadence default 24h, 60s startup grace before first backup. Combined with `skip_if_idle = true`, fresh-install deployments produce at most one ~30KB schema-only tarball and then stay quiet until there's real activity. Operators who want to opt out for throwaway dev/test set `enabled = false`. Asymmetry rationale: "wrong default on" costs ~30KB of disk on an unused test server; "wrong default off" costs everything on disk failure in production.
  - **Retention.** Two caps with OR semantics ŌĆö a tarball is deleted if it exceeds EITHER `retention_count` (default 10) OR `retention_age` (default 720h / 30 days). Each cap is a hard upper bound. Plan text initially read "AND"; corrected to OR during Step 5 implementation because OR matches operator intuition from every other retention system and "AND" would silently let `retention_count = 10` accumulate hundreds of tarballs as long as none crossed the age cap.
  - **`skip_if_idle` detection** compares (`PRAGMA data_version` on `data.db`, max mtime of `<dataDir>/data/files/`) against values captured at the last successful backup. Skip fires only if BOTH show no change. Baseline is in-memory only ŌĆö first tick after restart always runs (a restart often coincides with deployment changes worth snapshotting).
  - **Concurrent-backup mutex** (`backupMu sync.Mutex`, TryLock semantics) protects against scheduler-vs-scheduler and scheduler-vs-CLI-in-same-process collisions. Cross-process contention (server scheduler vs `sshkey-ctl backup` from another shell) is harmless ŌĆö different timestamps mean no filename collision and SQLite Online Backup tolerates concurrent readers.
  - **Lockfile for restore safety.** Server now writes `<dataDir>/sshkey-server.pid` on startup containing PID + start unix timestamp (atomic temp + rename); removed on graceful shutdown via `Server.Close`. Stale-PID detection via `syscall.Kill(pid, 0)` recovers from ungraceful shutdowns. New `internal/lockfile` package (importable by both server and CLI). Secondary value beyond restore: double-start protection against the same dataDir, reliable `sshkey-ctl status` running-vs-not reporting.
  - **`sshkey-ctl backup [--out PATH] [--label TAG]`** ŌĆö manual snapshot via the same code path the scheduler uses. Inherits `Compress` + `IncludeConfigFiles` + `DestDir` from `[backup]`; CLI flags override. Labels validated against `[A-Za-z0-9_-]{1,32}` at flag-parse time so path-traversal attempts (`--label ../../etc`) reject before any filesystem touch. `server.toml` is optional ŌĆö falls back to `DefaultServerConfig()` so operators can run backups against a copied dataDir on a recovery host.
  - **`sshkey-ctl restore <tarball> [--no-pre-backup]`** ŌĆö full 7-step flow with three independent safety nets:
    1. Lockfile check rejects with the live PID printed if the server is running. No filesystem touch.
    2. Tarball validation walks gzip + tar headers without extracting; rejects path-traversal, absolute paths, anything outside `data/` or `config/`, symlinks, non-regular-file entries, empty tarballs.
    3. Pre-restore backup prompt (default Y; `--no-pre-backup` skips). Non-TTY runs default to yes. Failure aborts restore unless opted out. The pre-restore tarball is a first-class restore point ŌĆö listable via `list-backups`, restorable via `sshkey-ctl restore`, archivable off-host via rsync.
    4. Free-space precheck via `syscall.Statfs` (heuristic: `tarballSize ├Ś 3`). Rejects with sizes printed if insufficient.
    5. Move current state to `<dataDir>/pre-restore-<YYYYMMDD-HHMMSS>/` safety directory (atomic `os.Rename` per artefact, EXDEV fallback to copy + remove for cross-filesystem moves). Skips source paths that don't exist locally.
    6. Extract tarball entry-by-entry (preserves file modes ŌĆö `host_key` stays 0600).
    7. `PRAGMA integrity_check` on every restored DB. Failure points operator at the safety dir for manual recovery.
  - **`sshkey-ctl list-backups`** ŌĆö newest-first table with NAME / SIZE / AGE / LABEL columns. Includes pre-restore tarballs (auto-labeled `pre-restore`) so operators can spot rollback candidates. Filename regex matches `backup-<YYYYMMDD-HHMMSS>[-<label>].tar.gz`; non-tarball files in `dest_dir` are silently ignored.
  - **`sshkey-ctl status` extended** with a `Backups:` block showing successes / failures / last success / last failure timestamps / last error message. Reads from `<dataDir>/.backup-stats.json` sidecar (atomic temp + rename writes from the scheduler; CLI reader tolerates missing file by reporting zeros).
  - **Process-state line in `sshkey-ctl status`** (lockfile-driven): reports `running (PID N) since <RFC3339 ts>`, `not running`, `not running (stale lockfile ŌĆö PID N exited)`, or `unknown (lockfile unreadable)`. The stale-lockfile case surfaces ungraceful crashes the operator should investigate.
  - **New packages**: `internal/lockfile` (PID-file primitives), `internal/backup` (tarball assembly + Online Backup wiring + retention).
  - **New config section** `[backup]` with 8 knobs (see `docker/config/server.toml` for the annotated reference). All knobs validated at startup with category-tagged errors (bad duration strings, negative counts, path-traversal in `dest_dir`).
  - **Driver dependency note:** `modernc.org/sqlite` v1.48.1 exposes Online Backup via `(*conn).NewBackup(dst) (*Backup, error)` accessed through `sql.DB.Conn(ctx).Raw(...)` + an anonymous-interface type assertion. Phase 19 is the first site that touches driver-specific features; the named import is narrowly scoped to `internal/backup/db.go`.
  - **~110 new tests** across all 7 implementation steps (config / lockfile / backup core / CLI backup / scheduler / CLI restore + list-backups). Includes end-to-end round-trip (backup ŌåÆ tamper marker row + delete attachment ŌåÆ restore ŌåÆ state verified restored) and rollback round-trip (restore ŌåÆ restore from pre-restore ŌåÆ original state restored).

- **Phase 17c: Error handling audit + typed error taxonomy + client send queue.** Completes the Phase 17 arc by wiring a typed error contract end-to-end across both repos and replacing the client's fire-and-forget send path with an in-memory queue + retry driver. Tagged **v0.2.0** ŌĆö the original Phase 17 tag target, preserved since this phase ships the reliability win the tag represents.
  - **Centralized error-response helper family** in `internal/server/reject.go`: `respondError(c, corrID, code, message, retryAfterMs)`, `respondOpaque(c, corrID)`, `respondErrorRef(c, corrID, code, ref)`, `respondUploadError`, `respondDownloadError`. ~104 scattered inline `c.Encoder.Encode(protocol.Error{...})` / `UploadError{...}` / `DownloadError{...}` sites migrated to the chokepoint family. Only 4 remaining `Encoder.Encode` sites for protocol.Error-family structs exist in the server package ŌĆö all inside the helper functions themselves. Uniform error shape is what makes the taxonomy enforceable.
  - **Four client-facing error categories** with `protocol.CategoryForCode(code)` as the authoritative classification function (test-enforced drift guard `TestCategoryForCode_ExhaustiveOverConstants` asserts exhaustive coverage over protocol error constants; parallel mirror in sshkey-term at `internal/protocol/categories.go`).
    - **Category A ŌĆö retriable transient** (`rate_limited`, `internal_error`, DB busy, broadcast back-pressure). Client auto-retries with exponential backoff (200msŌåÆ5s cap, max 5 attempts) then surfaces. A-default retries + surfaces on exhaustion (send / edit / react / admin verbs); A-silent refresh verbs (`room_members`, `device_list`) drop silently without user-visible error.
    - **Category B ŌĆö retriable after state fix** (`invalid_epoch`, `stale_member_list`). Server pushes fresh `epoch_key` or `group_list` alongside the error; client applies the pushed state and retries automatically. Per-(client, verb) throttle (~1s TTL) prevents 1000 invalid_epoch/sec from eliciting 1000 fresh state pushes.
    - **Category C ŌĆö permanent user-action** (`message_too_large`, `edit_window_expired`, `edit_not_most_recent`, malformed payload). Client surfaces to UI without retry.
    - **Category D ŌĆö privacy-identical rejection** (`unknown_room`, `unknown_group`, `non_member`, deleted-row). Byte-identical opaque error preserves the Phase 14 privacy invariant. Routed through `respondOpaque`.
  - **Correlation IDs end-to-end.** New `corr_<21-char-nanoid>` added as `CorrID string \`json:"corr_id,omitempty"\`` field on 15 client-to-server verb structs (`send` / `send_group` / `send_dm` / `edit` / `edit_group` / `edit_dm` / `react` / `unreact` / `delete` / `pin` / `unpin` / `room_members` / `device_list` / `history` / `download`). Echoed on all error responses and on 11 success response types (`Edited` / `GroupEdited` / `DMEdited` / `Reaction` / `ReactionRemoved` / `Deleted` / `HistoryResult` / `RoomMembersList` / `DeviceList` / `UploadReady` / `UploadComplete` / `DownloadStart`). `protocol.ValidateCorrID` shape-check feeds `SignalMalformedFrame` on violation. Server never persists corr_id ŌĆö pure client-private "which send was this" tag. Reuses existing `GenerateID("corr_")` helper; zero new dependency.

Each now fires `SignalMalformedFrame` + responds where wire shape supports it.
  - **5 pre-existing state-mutation partial-commit bugs fixed** alongside failure-injection tests that validate the fix:
    1. `handleUploadComplete` ŌĆö `StoreFileHash` return value was ignored; hash write failure left the row missing so subsequent downloads failed with `not_found`. Now aborts + `internal_error`.
    2. `handleSend` ŌĆö `InsertRoomMessage` error was logged but broadcast fired anyway; messages were visible to live recipients but missing from catchup on reconnect. Now aborts + `internal_error` + no broadcast.

Audit/history writes (~5 sites ŌĆö `RecordGroupEvent`, `RecordRoomEvent`, `RecordUserLeftRoom`, `audit.Log`, orphan-file cleanup) log-and-continue via `slog.Error`. - **Phase 17b: Auto-revoke on sustained misbehavior + connection hardening.** Automated device-revocation breaker that watches Phase 17's misbehavior counters and enqueues revocations for devices that sustain threshold crossings. Ships default-on; every auto-revoke signal has a zero legitimate baseline so well-behaved clients produce zero false positives.
  - **[server.auto_revoke] config** with `enabled` (default true), `prune_after_hours` (default 168), and `[server.auto_revoke.thresholds]` sub-table mapping signal names to `"count:window_seconds"` strings. Startup validation rejects unknown keys, load-category signals (`rate_limited`), and observational-category signals (`broadcast_dropped`) with category-tagged error messages. Data-driven: a new entry in `counters.AutoRevokeSignals` is auto-accepted by the config loader with no code change.
  - **Sliding-window counter API** (`counters.Check(signal, deviceID, threshold, windowSec)`). Per-entry ring buffer of event timestamps; `Inc` appends under a per-entry mutex. Ring cap default 64 (lazy-allocated ŌĆö idle entries carry no buffer memory). Package-internal `setNowFn` lets tests run deterministic-clock assertions without `time.Sleep`.
  - **Background processor** (`runAutoRevokeProcessor`, 10s cadence) ŌĆö iterates configured rules, calls `Check` per (signal ├Ś device), enqueues matches into the existing `pending_device_revocations` pipeline with `revoked_by = "server:auto_revoke"` and a human-readable reason string. No new tables, no new revocation code path ŌĆö existing `runDeviceRevocationProcessor` drains the queue.
  - **Observer mode** ŌĆö `enabled = false` keeps the goroutine running and evaluating thresholds; logs `auto_revoke_would_fire` at Info level instead of enqueuing. Operators diagnose false positives without disarming the breaker's observability.
  - **Write-path opportunistic pruning** + stale-filter on `Get` / `Snapshot` / `DevicesFor`. Per-entry `lastInc atomic.Int64` timestamp drives TTL-based eviction. Prune fires only on Inc's new-entry slow path when map size crosses an amortizing `nextPruneSize` threshold (starts at 64, doubles after each prune). Readers filter stale entries immediately ŌĆö callers never see stale data regardless of prune cadence.
  - **NDJSON idle-timeout watchdog** (Step 5a). Opt-in via `rate_limits.idle_timeout_seconds` (default 0 = disabled). Per-connection goroutine closes the SSH channel if no complete protocol frame arrives within the configured window. Slow-loris defense at the application layer; SSH-layer keepalive (30s) remains independently responsible for dead-TCP detection.
  - **Per-client outbound write queue + slow-reader disconnect** (Step 5b). `fanOut` now enqueues non-blocking into each recipient's per-connection `sendCh` (default 256 messages, configurable via `rate_limits.per_client_write_buffer_size`). A full queue counts `SignalBroadcastDropped`; after `rate_limits.consecutive_drop_disconnect_threshold` (default 10) consecutive drops to the same client, the channel closes (disconnect is the remedy ŌĆö client reconnects with clean state, no auto-revoke). Fixes the slow-recipient-blocks-fanout latency amplification inherent to the pre-Step-5b synchronous iteration. Writes go through a `safeEncoder` (mutex-wrapped `protocol.Encoder`) so the per-client writer goroutine and direct-Encode handler sites can safely share the SSH channel.
  - **Reconnect-flood signal** (Step 5c). New `counters.SignalReconnectFlood = "reconnect_flood"` added to `AutoRevokeSignals`. `handleSession` fires `Inc` on every successful session registration; combined with a `reconnect_flood = "10:60"` threshold in server.toml, a device thrashing session setup crosses threshold and auto-revokes. Note: keyed by `deviceID`, not SSH fingerprint ŌĆö the `sshkey-ctl block-fingerprint` CLI remains the manual remedy for attackers cycling device_ids under a single approved key.
  - **Reason-string UX contract** ŌĆö server owns the operator-readable string embedded in `device_revoked` events; format `"Automatic revocation: <description> (<count> events in <window>s)"`. TUI's `DeviceRevokedModel` renders verbatim ŌĆö client stays dumb, server holds the signal-to-description mapping (single source of truth in `auto_revoke.go`).
  - **Admin recovery is structural** ŌĆö a misconfigured breaker cannot trap an admin out. If an admin's device is auto-revoked, they retain OS-level SSH access to the server host and run `sshkey-ctl restore-device --user USER --device DEV` directly against the SQLite files. Flat thresholds for everyone; no admin-whitelist code in the server binary.
  - **Reference server.toml docs pass** ŌĆö `docker/config/server.toml` now documents every Phase 17 + 17b knob with inline trade-off comments. `TestDockerReferenceServerToml_LoadsCleanly` locks in that the reference file parses through the full Load + Validate pipeline.
  - **Ops runbook** ŌĆö `docs/ops/auto_revoke.md` covers monitoring, disable procedure, device restoration, and threshold tuning methodology.
  - **~110 new tests** across the counters package (sliding window, pruning, stale-filter) and server package (auto-revoke processor, idle watchdog, send queue, reconnect flood).

- **Phase 15: message editing.** Three new verb families on the wire ŌĆö `edit` / `edited` (rooms), `edit_group` / `group_edited` (group DMs), `edit_dm` / `dm_edited` (1:1 DMs). Each replaces the encrypted payload of an existing message row in place, preserving `id`, `sender`, `ts`, `file_ids`, and thread structure while setting a new server-authoritative `edited_at` on the broadcast echo. `Message`, `GroupMessage`, and `DM` protocol types gain an `EditedAt int64 \`json:"edited_at,omitempty"\`` field. Only the user's most recent non-deleted message in the current context is editable; non-members, non-authors, and deleted-row failure modes all collapse into byte-identical `ErrUnknownX` responses (privacy invariant). Authorship is surfaced via `ErrEditNotMostRecent`; room edits must be in the current-or-previous epoch grace window or they get `ErrEditWindowExpired`. Reactions on edited messages are cleared in the same transaction as the payload replace ŌĆö clients unconditionally drop local reaction state for the edited message ID on receipt of the `edited` event (no per-reaction `reaction_removed` broadcasts). New rate limit bucket `EditsPerMinute` (default 10/min, shared across all three verbs per user). New store helpers `UpdateRoomMessageEdited` / `UpdateGroupMessageEditedWithKeys` / `UpdateDMMessageEditedWithKeys` / `GetRoomMessageByID` / `GetGroupMessageByID` / `GetDMMessageByID` / `GetUserMostRecentMessageID{Room,Group,DM}`. New `edited_at INTEGER NOT NULL DEFAULT 0` column on the per-context `messages` schema (single CREATE TABLE edit via `initMessageDB`, no ALTER per the no-live-users policy). Byte-identical privacy regression tests: `TestHandleEdit_PrivacyResponsesIdentical`, `TestHandleEditGroup_PrivacyResponsesIdentical`, `TestHandleEditDM_PrivacyResponsesIdentical`. Plus happy-path, most-recent rejection, epoch-window rejection, retired-room rejection, deleted-row collapse, and reaction-clearing tests. See `PROTOCOL.md` "Message Editing" section for the wire format and `message_editing.md` for the design write-up.

### Security
- **New-member pre-join history gate for group DMs** ŌĆö new members added to an existing group DM no longer receive pre-join messages or pre-join `group_events` in their sync batches or history scroll-back. The wrapped-key crypto model already prevented DECRYPTION, but the server was still SERVING the rows, leaking timestamps, sender IDs, and audit-trail metadata (pre-join `/rename`, `/promote`, `/demote`, `/kick` events). `syncGroup` and `handleHistory`'s group branch now raise/post-filter by `group_members.joined_at`, mirroring the `first_seen/first_epoch` gate that has protected rooms since v0.1.0. The column existed since Phase 11 (`datetime('now')` default on `AddGroupMember`) but was never wired into the read paths ŌĆö fix is pure server-side, no schema or wire change. Covered by `TestSyncGroup_FiltersPreJoinMessages`, `TestSyncGroup_FiltersPreJoinEvents`, `TestSyncGroup_ReAddResetsJoinedAt`, `TestHandleHistory_FiltersPreJoinGroupMessages`, `TestSyncGroup_ExistingMemberSeesAllPostJoin`. See `groups_admin.md` "Pre-join history gate" section for the full write-up including the pins verification (rooms-only by design) and the 1:1 DM structural safety argument.

### Changed
- **Group DMs now have an in-group admin model (Phase 14)** ŌĆö reverses the "immutable peer DMs" decision from Phase 11. Group creators become the first admin; any admin can add/remove/promote/demote/rename. At-least-one-admin invariant enforced at every mutation path (`/leave`, `/delete`, demote, kick, retirement). Retirement cascade auto-promotes the oldest remaining member as successor. See `groups_admin.md` for the full design.
- **`handleRetirement` group branch restructured** ŌĆö replaces the bulk `RetireUserFromGroups` store helper with per-group iteration through `performGroupLeave(reason: "retirement", by: "")`. Fixes a latent orphan-on-solo bug where retiring a sole group member never triggered `DeleteGroupConversation`, leaving the `group_conversations` row + per-group DB file orbiting forever. `RetireUserFromGroups` and its 3 store-level tests are deleted.
- **`performGroupLeave` signature extended** from `(groupID, userID, reason string)` to `(groupID, userID, reason, by string)` so admin-initiated removals can render "You were removed from X by alice" instead of the generic "by an admin". `by` is empty for self-leave and retirement paths.
- **`handleRenameGroup` now requires admin** ŌĆö matches the byte-identical privacy convention (non-admin attempts collapse to `ErrUnknownGroup`). Pre-Phase-14 any member could rename.
- **`handleLeaveGroup` and `handleDeleteGroup` reject last-admin** with `ErrForbidden` when the group has other members. Sole-member carve-out: a user who is both the only member and the only admin can leave or delete freely (last-member cleanup runs).
- **Group retirement reason code** stays `"retirement"` (rooms use `"user_retired"`) ŌĆö 9+ sites across both repos depend on this distinction.
- **`performGroupLeave` now records an audit row** to the per-group `group_events` table before broadcasting, best-effort. Sole recording site for `leave` events per the audit contract.
