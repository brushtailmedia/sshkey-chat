- **Phase 21 F6 — lockfile TOCTOU race closed via tmpfile+Link (2026-04-19).** Pre-fix, `lockfile.Read()` checked if the existing lockfile's PID was alive and on stale/absent the caller proceeded to `lockfile.Write()` (Read-then-temp-write-then-Rename). Two `Server.New` calls racing could both pass the aliveness check in the microsecond window between reads, then both rename; the second clobbered the first's lockfile with both processes continuing to run. Severity reassessment 2026-04-19 downgraded this to LOW (realistic scenarios require deliberate operator misconfig; SQLite MVCC prevents corruption; worst case is brief split-brain until ops notices) — but upgraded to fix-now because the primitive swap was cheap. New implementation: stage the complete PID+timestamp payload in a per-process `.sshkey-lockfile-*.tmp` file in the same directory as the target, then call `os.Link(tmpName, path)` — POSIX link(2) is atomic and returns EEXIST if the target exists, making it the canonical exclusive-create primitive for file-based locks. Because the tempfile holds the complete payload before Link, readers always see complete content on the linked path; there is no empty-file or partial-content race that a naive `O_EXCL`-on-final-path approach would expose (we hit that in the first refactor attempt and caught it via the new concurrent test). Stale-recovery retries Link once after removing the stale file; concurrent stale-recovery collisions return `ErrAlreadyRunning` against the winning PID rather than looping. `defer os.Remove(tmpName)` cleans the staging file whether Link succeeded (inode persists via the linked path) or failed. New drift-guard test `TestWrite_ConcurrentAcquisitionExactlyOneSucceeds` runs 20 goroutines racing to acquire the same lockfile, deterministically asserts exactly one success and 19 ErrAlreadyRunning — pre-fix this would have been flaky or silently wrong; post-fix it's locked in by the atomic link(2) primitive. All 16 lockfile tests green; `go build ./... && go vet ./... && staticcheck ./...` clean; full `-race` suite green. See `docs/security/audit_v0.2.0.md#F6`.

- **Phase 19: Server backup & restore.** Server-side snapshot backups + retention + a stop-and-restore CLI. SQLite Online Backup API (via `modernc.org/sqlite`'s `(*conn).NewBackup`) snapshots every DB while the server keeps serving clients — no downtime, no client disconnects. Tarballs bundle SQLite DBs + attachment blobs + `audit.log` + `pending-keys.log` + `host_key` + `server.toml` (config files default-included; opt out via `[backup].include_config_files`). `rooms.toml` deliberately not included (seed file, ignored when `rooms.db` exists). See `docs/ops/backup_restore.md` for the operator runbook.
  - **Scheduled backups default-on.** `[backup] enabled = true` in the reference `server.toml`. Cadence default 24h, 60s startup grace before first backup. Combined with `skip_if_idle = true`, fresh-install deployments produce at most one ~30KB schema-only tarball and then stay quiet until there's real activity. Operators who want to opt out for throwaway dev/test set `enabled = false`. Asymmetry rationale: "wrong default on" costs ~30KB of disk on an unused test server; "wrong default off" costs everything on disk failure in production.
  - **Retention.** Two caps with OR semantics — a tarball is deleted if it exceeds EITHER `retention_count` (default 10) OR `retention_age` (default 720h / 30 days). Each cap is a hard upper bound. Plan text initially read "AND"; corrected to OR during Step 5 implementation because OR matches operator intuition from every other retention system and "AND" would silently let `retention_count = 10` accumulate hundreds of tarballs as long as none crossed the age cap.
  - **`skip_if_idle` detection** compares (`PRAGMA data_version` on `data.db`, max mtime of `<dataDir>/data/files/`) against values captured at the last successful backup. Skip fires only if BOTH show no change. Baseline is in-memory only — first tick after restart always runs (a restart often coincides with deployment changes worth snapshotting).

Only the user's most recent non-deleted message in the current context is editable; non-members, non-authors, and deleted-row failure modes all collapse into byte-identical `ErrUnknownX` responses (privacy invariant). Authorship is surfaced via `ErrEditNotMostRecent`; room edits must be in the current-or-previous epoch grace window or they get `ErrEditWindowExpired`. Reactions on edited messages are cleared in the same transaction as the payload replace — clients unconditionally drop local reaction state for the edited message ID on receipt of the `edited` event (no per-reaction `reaction_removed` broadcasts). New rate limit bucket `EditsPerMinute` (default 10/min, shared across all three verbs per user). New store helpers `UpdateRoomMessageEdited` / `UpdateGroupMessageEditedWithKeys` / `UpdateDMMessageEditedWithKeys` / `GetRoomMessageByID` / `GetGroupMessageByID` / `GetDMMessageByID` / `GetUserMostRecentMessageID{Room,Group,DM}`. New `edited_at INTEGER NOT NULL DEFAULT 0` column on the per-context `messages` schema (single CREATE TABLE edit via `initMessageDB`, no ALTER per the no-live-users policy). Byte-identical privacy regression tests: `TestHandleEdit_PrivacyResponsesIdentical`, `TestHandleEditGroup_PrivacyResponsesIdentical`, `TestHandleEditDM_PrivacyResponsesIdentical`. Plus happy-path, most-recent rejection, epoch-window rejection, retired-room rejection, deleted-row collapse, and reaction-clearing tests. See `PROTOCOL.md` "Message Editing" section for the wire format and `message_editing.md` for the design write-up.
